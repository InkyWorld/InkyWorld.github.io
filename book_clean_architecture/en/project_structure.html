Project Structure (DDD Style)

Folder organization is 50% of success. When you open a project, the structure should scream about what the system does, not what framework you are using.

Screaming Architecture

Robert Martin formulated the principle of Screaming Architecture: when looking at the top-level file structure, you should understand the purpose of the system.

If you open a Django project and see `models.py`, `views.py`, `urls.py` â€” this is a framework structure. It says nothing about the business. If you open a project and see `sales/`, `inventory/`, `billing/` â€” this is a domain-oriented structure.

Layers Breakdown

ğŸ¯ Domain Layer

The Heart of Business

Here live your Entities, Value Objects, and Domain Services. This code depends on nothing. No database refs, no HTTP, no JSON. Only pure Python and business rules.

  • Allowed: Standard library (`dataclasses`, `uuid`, `datetime`).
  • Forbidden: `django.db`, `sqlalchemy`, `requests`.
âš™ï¸ Application Layer

The Orchestrator (Use Cases)

This layer manages the flow of data. It receives commands from the presentation layer, loads entities via repository interfaces, calls domain methods, and saves results.

Here live Command Handlers, Query Handlers, and DTOs.

ğŸ”§ Infrastructure Layer

Implementation Details

Here interfaces defined in the Application Layer are implemented. This means databases, calls to external APIs, logging, email sending.

The Domain does not know about Infrastructure. The Infrastructure knows about the Domain.

ğŸŒ Presentation Layer

The Entry Doors

The entry point to the application. These can be REST API controllers (FastAPI, Flask), GraphQL resolvers, gRPC handlers, or even CLI commands.

The task of this layer is to accept an HTTP request, parse it, convert it to a DTO or Command, and pass it to the Application Layer.
