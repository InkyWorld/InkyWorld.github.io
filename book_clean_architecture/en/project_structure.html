Brief Overview of Each Component

  • domain/ â€” business rules: entities, value objects, domain services, domain events. No external dependencies.
  • application/ â€” use cases, ports (interfaces), and orchestration. Repository and UoW interfaces are defined here.
  • infrastructure/ â€” adapter implementations: ORM, message brokers, external API clients. Depends on application/domain.
  • presentation/ â€” API endpoints, CLI, background task entrypoints. Composes the application from adapters and services.
  • tests/ â€” unit and integration tests. Prefer testing domain & application in isolation using fakes/mocks for infrastructure.

Practical Tips

  • Domain â€” pure code without external library imports (except typing, dataclasses, and stdlib).
  • Application stores interfaces (ports); Infrastructure implements them (adapters).
  • Use UoW to ensure atomicity of operations (transactions).
  • Packages should be small and clear; tests alongside code or in a separate `tests/` directory.
