Versioning

Optimistic Locking and conflict resolution when saving aggregates.

The Concurrency Problem

Two managers, Alice and Bob, opened the same Order simultaneously. Alice moved it to the "Paid" status, while Bob, unaware of this, added an item. If Bob saves the order after Alice, he will overwrite her changes (the so-called Lost Update Problem).

Danger: We might end up with a paid order containing new unpaid items, violating the consistency invariant.

Optimistic Concurrency Control (OCC)

The solution is to add a version (int) field to the Aggregate.

  • When loading the aggregate, we remember `version_at_load = 5`.
  • When saving, we tell the DB:
    UPDATE orders SET ..., version = 6 WHERE id = X AND version = 5.
  • If the version changed to 6 in the meantime, our query will update zero rows.
  • The Application Layer catches this and either throws an error to the user or reloads the aggregate and tries again (retry).

Implementation Example

@dataclass
class Order(AggregateRoot):
    id: str
    items: List[Item]
    # Technical field for OCC
    _version: int = 0

    def add_item(self, item):
        self.items.append(item)
        # Note: we do NOT increment the version manually here.
        # This is the Repository's responsibility upon saving.

# In Repository (Infrastructure)
def save(self, order: Order):
    current_version = order._version
    new_version = current_version + 1
    
    result = db.execute(
        """
        UPDATE orders 
        SET data = :data, version = :new_v 
        WHERE id = :id AND version = :old_v
        """,
        {"id": order.id, "data": json(order), "new_v": new_version, "old_v": current_version}
    )
    
    if result.rowcount == 0:
        raise ConcurrentUpdateException(f"Order {order.id} was modified by someone else")
        
    order._version = new_version

In Event Sourcing

In Event Sourcing, the ExpectedVersion field plays the same role. When you try to append events to a stream, you say: "I expect there to be 10 events in the stream right now". If there are already 11 (someone added an event), the Event Store will reject the write.
