Invariants and Domain Errors

How to ensure your application never reaches an "impossible" state.

1. What is an Invariant?

Invariant â€” is a condition that must always remain true for a system or object to be considered valid and consistent.

These are the laws of physics of your world. If your world dictates "One cannot spend more than they have in their account" â€” that is an invariant. If it is violated, the program is broken.

Real-life Example: An elevator's invariant â€” "Doors cannot be open while the elevator is moving". If this rule is violated even for a millisecond, a catastrophe will occur.

Where to check them?

  • If the rule concerns data format (email must have @) â€” it belongs in a Value Object.
  • If the rule concerns state changes (cannot pay twice) â€” it belongs in an Entity.
  • If the rule concerns uniqueness in DB (email must be unique) â€” it belongs in a Domain Service or a Repository check (called from a Use Case).

2. Error Architecture (Domain Exceptions)

Never use built-in `ValueError`, `TypeError`, or `Exception` to describe business problems. Create your own error classes.

Why?

  1. Readability: `raise InsufficientFundsError` says more than `raise ValueError("Not enough money")`.
  2. Handling: In the Presentation Layer (FastAPI), you can catch specific errors and return a proper 400 or 409 response to the user.
# domain/exceptions.py

# 1. Base class. Inherit from this.
class DomainError(Exception):
    """Base business logic error"""
    pass

# 2. Value Object Errors
class InvalidEmailError(DomainError):
    pass

class NegativeMoneyError(DomainError):
    pass

# 3. Entity Errors
class OrderAlreadyPaidError(DomainError):
    pass

class OrderEmptyError(DomainError):
    pass

3. Validation in Value Objects

A Value Object must be "always valid". It is impossible to create a `Money(-100)` object. Validation must happen at the moment of creation.

In Python with `dataclasses`, the ideal place for this is the `__post_init__` method.

from dataclasses import dataclass
from .exceptions import NegativeMoneyError

@dataclass(frozen=True)
class Money:
    amount: float
    currency: str = "USD"

    def __post_init__(self):
        # INVARIANT: Money >= 0
        if self.amount < 0:
            raise NegativeMoneyError(f"Amount cannot be {self.amount}")

# Usage:
# m = Money(100)  # OK
# m = Money(-50)  # Exception raised instantly!
Pattern: Parse, don't validate.
Do not write functions like `is_email_valid(str) -> bool`. Instead, create an object `Email(str)`. If the creation succeeds â€” the data is valid. If not â€” an error is raised.

4. Validation in Entities (State Consistency)

Entities live in time. Their invariants often depend on the current state.

We use "Guard Clauses" at the beginning of methods.

class Order:
    def __init__(self, ...):
        self._is_paid = False
        self._items = []

    def pay(self):
        # INVARIANT 1: Cannot pay twice
        if self._is_paid:
            raise OrderAlreadyPaidError("Order is already paid")

        # INVARIANT 2: Cannot pay for an empty order
        if len(self._items) == 0:
            raise OrderEmptyError("Cannot pay for an empty order")

        # If checks pass - change state
        self._is_paid = True

    def add_item(self, item):
        # INVARIANT 3: Freeze composition after payment
        if self._is_paid:
            raise OrderAlreadyPaidError("Cannot modify items of a paid order")
        
        self._items.append(item)

âš ï¸ Validation Anti-patterns

1. External Validation Service

Creating an `OrderValidator` class that checks the order from the outside.

validator.validate(order) # BAD!

Why it's bad: Encapsulation is violated. Order rules must live inside the Order. If you add a new field to the Order, you will forget to update the validator.

2. HTTP Exceptions in Domain

Importing `HTTPException` from FastAPI inside an Entity.

raise HTTPException(status_code=400) # BAD!

Why it's bad: Your domain now depends on the Web Framework. You won't be able to use this code in a CLI or Telegram bot. Throw `DomainError`, and the controller (Presentation Layer) will convert it to a 400.

3. Returning False instead of Error

if error: return False # BAD!

Why it's bad: You lose context (why did the error happen?) and force the calling code to guess. Exceptions are not evil, they are a means of communication.
