Theory

SOLID Principles in Python

Five laws of flexible code. If your code is hard to test or change, you've likely violated one of them.

S

Single Responsibility Principle (SRP)

"A class should have one, and only one, reason to change."

Don't confuse this: it does not mean a class should have one method. It means a class should be responsible for one business function.

‚ùå God Object (Anti-pattern)
class Report:
    def calculate_stats(self):
        # Calculation logic
        pass

    def generate_pdf(self):
        # PDF Generation Logic
        pass

    def save_to_file(self):
        # Filesystem operations
        pass
‚úÖ Separation of Concerns
class StatsCalculator:
    def calculate(self): ...

class PdfReporter:
    def render(self, data): ...

class FileSystemSaver:
    def save(self, content): ...
Rookie Mistake:

Thinking that SRP forbids adding helper methods.
Example: In a `User` class, `get_full_name()` is fine. But `save_to_db()` is a violation of SRP because it mixes data with persistence.

O

Open/Closed Principle (OCP)

"Software entities should be open for extension, but closed for modification."

You should be able to add a new feature (e.g., a new payment method) without changing old working code.

‚ùå Bad: If/Elif Hell

class PaymentProcessor:
    def pay(self, amount, type):
        if type == "credit_card":
            # Card Logic
            pass
        elif type == "paypal":
            # PayPal Logic
            pass
        # To add Bitcoin, you have to modify this file and risk breaking cards!

‚úÖ Good: Polymorphism

from abc import ABC, abstractmethod

class PaymentMethod(ABC):
    @abstractmethod
    def pay(self, amount):
        pass

class CreditCard(PaymentMethod):
    def pay(self, amount): ...

class PayPal(PaymentMethod):
    def pay(self, amount): ...

# Adding Bitcoin does not touch old code:
class Bitcoin(PaymentMethod):
    def pay(self, amount): ...
L

Liskov Substitution Principle (LSP)

"Subtypes must be substitutable for their base types."

If your code expects an `Animal`, it should work correctly with both `Dog` and `Cat`. If `Dog` suddenly throws an error where `Animal` shouldn't ‚Äî that is a violation.

Classic Example: Square is not a Rectangle

In geometry, a square is a special case of a rectangle. In programming ‚Äî no! If you change the width of a rectangle, the height remains. For a square ‚Äî no.

Real Life Example: ReadOnly Repository

class Repository:
    def save(self, obj): ...

class ReadOnlyRepository(Repository):
    def save(self, obj):
        # VIOLATION of LSP!
        # Client code expects save() to work, but gets an error.
        raise NotImplementedError("I am read-only")
Solution: Split the hierarchy into `Readable` and `Writable`.
I

Interface Segregation Principle (ISP)

"Clients should not be forced to depend upon methods that they do not use."

It is better to have many small interfaces than one huge one (Fat Interface).

‚ùå One Fat Interface

class SmartDevice(ABC):
    def print(self): ...
    def scan(self): ...
    def fax(self): ...

# If I have an old printer, 
# I still have to implement 
# scan() and fax(), even if empty.

‚úÖ Segregated Interfaces

class Printer(ABC):
    def print(self): ...

class Scanner(ABC):
    def scan(self): ...

# Now we can compose an MFP 
# via multiple inheritance
class MFP(Printer, Scanner): ...
D

Dependency Inversion Principle (DIP)

"Depend upon abstractions, not concretions."

This is the foundation of Clean Architecture. A Use Case should not know about Postgres. It should know about an abstract `Repository`.

# ‚ùå VIOLATION of DIP: Hard dependency
class CreateOrderService:
    def __init__(self):
        # We nailed down a specific SQL DB implementation!
        # Now this code is impossible to test without a real DB.
        self.repo = SqlAlchemyRepository() 

# ‚úÖ ADHERENCE to DIP: Dependency Injection
class CreateOrderService:
    # We ask for "some repository" (Abstraction)
    def __init__(self, repo: OrderRepository):
        self.repo = repo
Rookie Mistake:

Using `import` inside methods or functions to "hide" the dependency. This doesn't solve the problem; the dependency is still hard-coded, just harder to find.
