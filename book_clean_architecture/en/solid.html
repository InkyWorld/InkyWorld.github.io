SOLID Principles in Python

Five laws of flexible code. If your code is hard to test or changeâ€”you've likely violated one of them.

S

Single Responsibility Principle (SRP)

"A class should have only one reason to change."

Don't confuse this: it doesn't mean a class should have only one method. It means a class should be responsible for one business function.

âŒ God Object (Antipattern)
class Report:
    def calculate_stats(self):
        # Calculation logic
        pass

    def generate_pdf(self):
        # PDF rendering logic
        pass

    def save_to_file(self):
        # File system operations
        pass
âœ… Separation of Concerns
class StatsCalculator:
    def calculate(self): ...

class PdfReporter:
    def render(self, data): ...

class FileSystemSaver:
    def save(self, content): ...
Beginner Mistake:

Thinking that SRP prohibits adding helper methods.
Example: A `get_full_name()` method in a `User` class is fine. But `save_to_db()` violates SRP, as it mixes data with persistence (storage).

O

Open/Closed Principle (OCP)

"Software entities should be open for extension, but closed for modification."

You should be able to add a new feature (e.g., a new payment method) without changing existing working code.

âŒ Bad: If/Elif Hell

class PaymentProcessor:
    def pay(self, amount, type):
        if type == "credit_card":
            # Credit card logic
            pass
        elif type == "paypal":
            # PayPal logic
            pass
        # To add Bitcoin, you'd have to modify this file and risk breaking cards!

âœ… Good: Polymorphism

from abc import ABC, abstractmethod

class PaymentMethod(ABC):
    @abstractmethod
    def pay(self, amount):
        pass

class CreditCard(PaymentMethod):
    def pay(self, amount): ...

class PayPal(PaymentMethod):
    def pay(self, amount): ...

# Adding Bitcoin doesn't touch existing code:
class Bitcoin(PaymentMethod):
    def pay(self, amount): ...
L

Liskov Substitution Principle (LSP)

"Subtypes must be substitutable for their base types."

If your code expects an `Animal`, it should work correctly with both `Dog` and `Cat`. If `Dog` suddenly throws an error where `Animal` shouldn'tâ€”that's a violation.

Classic Example: Square is not a Rectangle

In geometry, a square is a special case of a rectangle. In programmingâ€”no! If you change the width of a rectangle, the height stays the same. For a squareâ€”it doesn't.

Real-World Example: ReadOnly Repository

class Repository:
    def save(self, obj): ...

class ReadOnlyRepository(Repository):
    def save(self, obj):
        # LSP VIOLATION!
        # Client code expects save() to save, but gets an error.
        raise NotImplementedError("I'm read-only")
Solution: Split the hierarchy into `Readable` and `Writable`.
I

Interface Segregation Principle (ISP)

"Clients should not be forced to depend on methods they don't use."

Many small interfaces are better than one huge (Fat Interface).

âŒ Fat Interface

class SmartDevice(ABC):
    def print(self): ...
    def scan(self): ...
    def fax(self): ...

# If I have an old printer, 
# I still have to implement 
# scan() and fax(), even if empty.

âœ… Segregated Interfaces

class Printer(ABC):
    def print(self): ...

class Scanner(ABC):
    def scan(self): ...

# Now you can build an MFP via 
# multiple inheritance
class MFP(Printer, Scanner): ...
D

Dependency Inversion Principle (DIP)

"Depend on abstractions, not on concretions."

This is the foundation of Clean Architecture. Use Case shouldn't know about Postgres. It should know about an abstract `Repository`.

# âŒ DIP VIOLATION: Hard dependency
class CreateOrderService:
    def __init__(self):
        # We've nailed down a specific SQL database implementation!
        # Now this code can't be tested without a real database.
        self.repo = SqlAlchemyRepository() 

# âœ… DIP COMPLIANCE: Dependency injection
class CreateOrderService:
    # We ask for "some repository" (Abstraction)
    def __init__(self, repo: OrderRepository):
        self.repo = repo
Beginner Mistake:

Using `import` inside methods or functions to "hide" the dependency. This doesn't solve the problemâ€”the dependency is still hard-coded, it's just harder to find.
