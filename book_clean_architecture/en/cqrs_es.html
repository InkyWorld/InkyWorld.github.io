CQRS and Event Sourcing

Divide and Conquer. Advanced patterns for scalable systems.

1. CQRS (Command / Query Responsibility Segregation) â€” In Brief

TL;DR: We separate the write path (Commands) from the read path (Queries). Writing focuses on consistency and transactions â€” making it more complex; reading focuses on speed and aggregated projections â€” making responses fast.

Why is this important

In a real system, requirements for reading and writing often conflict: writing must preserve invariants and be atomic, while reading must return data in milliseconds and support complex aggregates/reports. CQRS allows optimizing each side separately.

How it looks â€” step by step

  1. Command (Write): The Use Case/Service handles the command, applies business rules, and saves changes (usually a normalized model + Outbox for emitting events).
  2. Event: After a successful save, a Domain Event is created and placed in the Outbox / Event Store.
  3. Projection (Read Model): A background worker reads events and updates denormalized projections (summary tables) that are optimized for reading.
  4. Query: API/Query Handlers read from these projections, responding quickly and efficiently.

When to use / When not to

  • Use when: high read load, complex views/reports, audit/history requirements, need for horizontal scaling of reads.
  • Don't use when: simple CRUD application with low traffic â€” the overhead of CQRS/ES is not worth it.

Glossary

  • Domain Event â€” A fact that has already happened in the past ("Order Paid"). It is immutable.
  • Outbox Pattern â€” A way to guarantee message delivery. Instead of sending directly to RabbitMQ (which might fail), we save the event in a local outbox database table within the same transaction as business data. Then a separate background process reads this table and sends messages. This guarantees At-Least-Once Delivery.
  • Projection / Read Model â€” A denormalized table "tailored" for a specific UI screen. It is updated asynchronously upon receiving events. For example, instead of a complex JOIN query across 5 tables, we pre-calculate a flat OrdersSummary table, from which the API reads data instantly.
  • Snapshot â€” Optimization for Event Sourcing. To avoid replaying 1,000,000 events from the beginning of time to get the current state of an aggregate, we periodically save a "slice" of it (e.g., every 100th version). When loading, we take the last Snapshot and apply only new events.

ğŸ’¡ Tip: start simple â€” separate Read/Write (CQRS) without Event Sourcing, then introduce ES only when necessary.

2. Domain Events

An event is a fact that already happened. "Order Paid", "User Registered". Events allow reducing coupling between modules.

3. Event Sourcing (ES)

Core Idea: We don't store the current state of the object (a row in the DB). We store the history of all events that led the object to this state.

Traditional Approach (State-Oriented)

User(id=1, balance=500)
â¬‡ UPDATE balance=600
User(id=1, balance=600)
History lost. Where did 100 come from?

Event Sourcing

Ev1: UserCreated(balance=0)
Ev2: MoneyDeposited(1000)
Ev3: ItemPurchased(500)
-------------------------
Current State = sum(events) = 500

How does it work?

  1. Event Store: This is an Append-Only database. We never delete or modify events.
  2. Replay: To get the `Order` object, we take an empty order and "apply" all events from the database in order.
  3. Projections: Restoring an object from 1000 events every time is slow. Therefore, we listen to events in the background and update regular SQL tables ("Read Models") for fast searching.
class Order(AggregateRoot):
    # Restore state (Apply)
    def apply(self, event: DomainEvent):
        if isinstance(event, OrderCreated):
            self.status = "NEW"
            self.total = 0
        elif isinstance(event, ItemAdded):
            self.total += event.price

    # Load from history
    @classmethod
    def load_from_history(cls, events: List[DomainEvent]):
        order = cls()
        for event in events:
            order.apply(event)
        return order

Pros and Cons

  • Audit Log: Solid audit out of the box. You cannot "quietly" fix the database.
  • Temporal Query: "What was the user's balance last Friday?" â€” Easy, just replay events up to that date.
  • Complexity: Event versioning (Data Migration for events is hell). Eventually Consistency (Projection lag).

4. Common Mistakes and Antipatterns

  • Mixing Read/Write models: attempts to use the same data structure for high-load queries and transactional consistency lead to performance compromises.
  • Ignoring idempotency: event handlers must be idempotent â€” delivery is usually at-least-once.
  • Lack of metadata: without sequence/correlation/cause, it is difficult to debug and maintain order and causal relationships.
  • Poor event migration strategy: changing event schema without clear versioning and migrations leads to inability to correctly replay history.
  • Direct projection modification: manually editing the Read Model is a fast track to desynchronization; update projections via events.

5. Lifehacks and Best Practices

  • Outbox Pattern: write the event to the Outbox within the same transaction as the state change â€” this guarantees atomicity and simplifies integration with brokers.
  • Add metadata: store aggregate_id, sequence, correlation_id, causation_id, and schema_version in each event.
  • Idempotent handlers: keep a set of processed event_ids or use transactions with a unique key to prevent duplicate processing.
  • Snapshots: for aggregates with long history, make periodic snapshots of the state â€” speeds up loading.
  • Start with CQRS without ES: often separating Read/Write paths is enough, and it gives most benefits without the complexity of Event Sourcing.
  • Version events: when changing, create a new version of the event and migrate handlers gradually.

7. Testing and Debugging

  • Recovery tests: verify that replaying events leads to expected aggregate state.
  • Projection tests: simulate event stream and check Read Models.
  • In-memory E2E: run Event Store in memory for integration tests (more like component tests).
  • Debugging tools: ability to "scroll" state to a specific event and see diffs is priceless.

9. Exercises

  1. Exercise 1 (Medium): Implement Outbox
    • Add an Outbox table and modify aggregate saving code to also save serialized events to Outbox in one transaction.
    • Implement a background worker that reads Outbox and publishes events to RabbitMQ/Kafka, marking records as sent.
    • Acceptance: events are not lost on rollback; upon worker restart, events are sent exactly once (can use sent_at flag).
  2. Exercise 2 (Medium): Projection and Test
    • Create an orders_summary projection that stores total_amount and status for fast reading.
    • Write a unit test that feeds a series of events (OrderCreated, ItemAdded, OrderPaid) and checks projection state.
    • Acceptance: projection matches the expected state after the sequence of events.
  3. Exercise 3 (Hard): Idempotent Handler and DLQ
    • Make a handler that writes payment_id to processed_events table, preventing re-processing.
    • Add a mechanism for sending "failed" messages to DLQ after N attempts; write a test simulating permanent failure.
    • Acceptance: re-delivery does not lead to duplicates; DLQ contains record after limit exceeded.
  4. Exercise 4 (Hard): Snapshotting
    • Implement snapshot approach to speed up loading of aggregate with very long history (save state snapshot every N events).
    • Add a test checking that recovery from snapshot + subsequent events yields correct state.

10. Hints and Solutions

  • Outbox: move message sending out of the main thread to a separate process; do not do send inside commit code.
  • Idempotency: use unique indexes on event_id or message_id to protect against duplicates.
  • Testing: for integration tests, use docker-compose with in-memory broker and database to speed up scenario verification.
