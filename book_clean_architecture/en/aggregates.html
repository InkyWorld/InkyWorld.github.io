Aggregates

Transaction boundaries and data consistency.

Aggregate â€” is a cluster of objects (Entities and Value Objects) that we treat as a single unit for data changes. It is a consistency boundary.

Analogy: Car

Imagine a car. It has wheels, an engine.
Wheels are entities, but they make no sense without the car. You cannot simply "tighten" a wheel without considering the state of the entire car (is it moving or parked).
Car â€” is the Aggregate Root. All commands (drive, turn) go through the steering wheel (Root), not directly to the wheels.

Core Rules

1. One Root

An Aggregate has only one main entity â€” the Root. The outside world (other services, repositories) can refer only to the Root. Internal objects (OrderLine) are hidden from the outside world.

2. Transaction Boundary

One database transaction should modify only one aggregate instance. If you need to change two aggregates, use Eventual Consistency (Domain Events).

3. Reference by ID

Aggregates reference each other only by ID (`customer_id`), not by direct object references (`self.customer`). This allows for easy separation in memory and database.

4. Invariants

The Root is responsible for the validity of the entire object graph. For example, the `Total` of an order must always equal the sum of `LineItems`. You cannot add a line without updating the Total.

Code Example

Note: `OrderLine` is created and lives only within `Order`.

@dataclass
class OrderLine:
    sku: str
    price: Money
    qty: int

@dataclass
class Order(AggregateRoot): # Root
    id: str
    _lines: List[OrderLine] = field(default_factory=list) # Internal entities
    _status: str = "NEW"
    total_amount: Money = Money(0, "USD")

    def add_product(self, sku: str, price: Money, qty: int):
        '''
        Root method - the only entry point for changes.
        Ensures the order total is always current.
        '''
        if self._status != "NEW":
            raise DomainException("Cannot modify a confirmed order")

        # Logic for working with internal entities
        existing_line = next((l for l in self._lines if l.sku == sku), None)
        if existing_line:
            existing_line.qty += qty
        else:
            self._lines.append(OrderLine(sku, price, qty))
        
        # Recalculate invariant (total amount)
        self._recalculate_total()

    def _recalculate_total(self):
        # Private method, called on any change
        self.total_amount = sum(l.price * l.qty for l in self._lines)

Common Mistake: Giant Aggregates

Do not make an aggregate "for the whole world". If a `Shop` aggregate contains a list of all `Product` (thousands of objects), then:

  1. When loading Shop, you load all 10,000 products into memory.
  2. Two managers cannot edit different products simultaneously because they lock the same Shop (Optimistic Lock error).

Solution:

Split into smaller aggregates. `Shop` separate, `Product` separate. Reference by ID.
