Aggregates

Transactional boundaries and data consistency.

Aggregate is a cluster of objects (Entities and Value Objects) that we treat as a single unit for data changes. It's a consistency boundary.

Analogy: Automobile

Imagine a car. It has wheels, an engine.
Wheels are entities, but they don't make sense without the car. You can't just "tweak" a wheel without considering the state of the entire car (is it moving or parked).
Car is the Aggregate Root. All commands (accelerate, turn) go through the steering wheel (Root), not directly to the wheels.

Core Rules

1. Single Root

An aggregate has only one main entity â€” the Root. The external world (other services, repositories) can reference only the Root. Internal objects (OrderLine) are hidden from the outside.

2. Transactional Boundary

A single database transaction should modify only one instance of an aggregate. If you need to modify two aggregates â€” use Eventual Consistency (Domain Events).

3. References by ID

Aggregates reference each other only by ID (`customer_id`), not direct object references (`self.customer`). This allows easy separation in memory and database.

4. Invariants

The Root is responsible for validity of the entire object graph. For example, order `Total` should always equal the sum of `LineItems`. You can't add a line without updating Total.

Code Example

Note: `OrderLine` is created and lives only inside `Order`.

@dataclass
class OrderLine:
    sku: str
    price: Money
    qty: int

@dataclass
class Order(AggregateRoot): # Root
    id: str
    _lines: List[OrderLine] = field(default_factory=list) # Internal entities
    _status: str = "NEW"
    total_amount: Money = Money(0, "USD")

    def add_product(self, sku: str, price: Money, qty: int):
        '''
        Root's method - the only entry point for changes.
        Guarantees that order total is always up-to-date.
        '''
        if self._status != "NEW":
            raise DomainException("Cannot modify a confirmed order")

        # Logic for working with internal entities
        existing_line = next((l for l in self._lines if l.sku == sku), None)
        if existing_line:
            existing_line.qty += qty
        else:
            self._lines.append(OrderLine(sku, price, qty))
        
        # Recalculate invariant (total amount)
        self._recalculate_total()

    def _recalculate_total(self):
        # Private method, called on any change
        self.total_amount = sum(l.price * l.qty for l in self._lines)

Common Mistake: Huge Aggregates

Don't make an aggregate "for the entire world". If a `Shop` aggregate contains a list of all `Product` (thousands of objects), then:

  1. Loading Shop will load all 10,000 products into memory.
  2. Two managers cannot simultaneously edit different products, as they lock the same Shop (Optimistic Lock error).

Solution:

Split into smaller aggregates. `Shop` separately, `Product` separately. Reference by ID.
