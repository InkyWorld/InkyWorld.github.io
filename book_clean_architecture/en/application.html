Orchestration

1. Application Layer

This is the conductor of your orchestra. It doesn't play the instruments (business logic), but it instructs who should play and when.

What is the Application Layer?

This layer implements the system's Use Cases. It answers the question "What does this application do?", rather than "How is it implemented?".

Primary Responsibility

Take data from UI → Load aggregate from DB → Call method on aggregate → Save changes → Send email.

The Application Layer knows about the Domain and Infrastructure interfaces (abstractions), but does not know about HTTP, JSON, SQL, or other implementation details.

2. Use Case Example

Let's look at a "Create Order" scenario.

from dataclasses import dataclass

# DTO (Input): Simple data structure.
# No logic, only what came from the user.
@dataclass
class CreateOrderDTO:
    user_id: int
    item_ids: list[int]

class CreateOrderUseCase:
    # Dependencies are injected (DI) via constructor.
    # We work ONLY with interfaces (abstractions).
    def __init__(
        self, 
        repo: OrderRepository,        # Interface to DB
        warehouse: WarehouseService,  # Interface to another service
        uow: UnitOfWork               # Transactional consistency
    ):
        self.repo = repo
        self.warehouse = warehouse
        self.uow = uow

    def execute(self, data: CreateOrderDTO) -> int:
        # 1. Orchestration: Get data (Infrastructure)
        if not self.warehouse.has_stock(data.item_ids):
             raise OutOfStockError()

        # 2. Execute business rules (Domain)
        # We ask the Domain to create the order. The logic "how" is there.
        order = Order.create(
            user_id=data.user_id, 
            items=data.item_ids
        )

        # 3. Save state (Infrastructure)
        self.repo.add(order)
        
        # 4. Commit transaction
        self.uow.commit()

        # 5. Return result
        return order.id

3. Commands and Queries (CQS)

Use cases are often divided into two categories:

Command

Changes State

CreateOrder, UpdateUser, PayInvoice.

  • Has side effects (modifies DB).
  • Operates on Domain Models.
  • Usually returns nothing or an ID.
Query

Returns Data

GetOrderById, ListRecentUsers.

  • No side effects (read-only).
  • Can return DTOs directly from DB (bypassing domain!) for speed.

4. DTO & Commands

DTO (Data Transfer Object) is a box for transferring data. It has no behavior.

The external world (Web API) sends us JSON (primitives: strings, numbers). The Domain works with complex objects (Money, Email). The Application Layer must accept primitives via DTO and convert them to Domain objects.

Command vs Query:
  • Command: DTO that changes something (`CreateOrderDTO`).
  • Query: DTO for retrieving data (`GetOrderDTO`).
# application/dto.py
from dataclasses import dataclass
from typing import List

# Input DTO (Command)
@dataclass(frozen=True)
class CreateOrderCommand:
    customer_email: str
    items: List[dict] # [{'name': 'Book', 'price': 10.5}]

    def __post_init__(self):
        # Simple input validation (not business rules!)
        if not self.customer_email:
            raise ValueError("Email is required")
        if not self.items:
            raise ValueError("Items list cannot be empty")

# Output DTO (Result)
@dataclass(frozen=True)
class OrderResultDTO:
    order_id: str
    total_amount: float
    status: str
Query DTO in practice:

Query changes nothing ‚Äî it just reads data and returns it in a convenient format.

# application/dto.py (Query part)
@dataclass(frozen=True)
class GetOrderQuery:
    order_id: str

# Output DTO for Query response
@dataclass(frozen=True)
class OrderDetailsDTO:
    order_id: str
    customer_email: str
    items: List[dict]
    total_amount: float

Query Use Case receives `GetOrderQuery`, reads entity through port and returns `OrderDetailsDTO`, without changing state.

5. Ports (Interfaces)

Dependency Inversion

Use Case wants to save an order. But it cannot call `SQLAlchemy`. So it declares a Port (Interface): "I need someone who can save orders".

Later, in the Infrastructure layer, we implement this interface. This allows Use Case to depend on abstraction, not on details.

# application/interfaces.py
from abc import ABC, abstractmethod
from typing import Optional
import uuid

# We import Entity because Port works with Domain
from domain.entities import Order 

class OrderRepository(ABC):
    """Abstract repository class. No SQL here!"""
    
    @abstractmethod
    def save(self, order: Order) -> None:
        pass

    @abstractmethod
    def get_by_id(self, oid: uuid.UUID) -> Optional[Order]:
        pass

class EmailService(ABC):
    @abstractmethod
    def send_order_confirmation(self, email: str, order_id: str):
        pass

6. Use Cases (Scenario Implementation)

This is the most important code in this layer. It ties everything together.

Use Case Algorithm:

  1. Accept DTO (Command).
  2. Convert primitives to Value Objects (str -> Email).
  3. Load/Create Entity (or several).
  4. Call Entity methods (all business logic executes THERE).
  5. Save Entity through Repository.
  6. Return result (Output DTO).
# application/use_cases.py
import uuid
from domain.entities import Order
from domain.value_objects import Money, Email
from .interfaces import OrderRepository, EmailService
from .dto import CreateOrderCommand, OrderResultDTO

class CreateOrderUseCase:
    # DEPENDENCY INJECTION:
    # We ask for any implementations of these interfaces.
    def __init__(
        self, 
        repo: OrderRepository, 
        email_service: EmailService
    ):
        self.repo = repo
        self.email_service = email_service

    def execute(self, command: CreateOrderCommand) -> OrderResultDTO:
        # 1. Prepare Value Objects
        # If email is incorrect, error will be raised HERE (invariant protection)
        email_vo = Email(command.customer_email)
        order_id = uuid.uuid4()

        # 2. Create Entity
        order = Order(order_id, email_vo)

        # 3. Execute business actions
        for item in command.items:
            price_vo = Money(item['price'])
            # All logic "can item be added" is inside order.add_item()
            order.add_item(item['name'], price_vo)

        # 4. Save through Port
        self.repo.save(order)

        # 5. Side effects (notifications)
        self.email_service.send_order_confirmation(
            email=email_vo.value, 
            order_id=str(order.id)
        )

        # 6. Return result (Primitives only!)
        return OrderResultDTO(
            order_id=str(order.id),
            total_amount=order.total_price.amount,
            status="created"
        )

7. Orchestrating Multiple Entities

Often you need two entities to interact. For example: "When creating an order, if User has VIP status, apply discount".

Who does this? Use Case. It's the conductor.

# application/use_cases.py (extended example)

def execute(self, command: CreateOrderCommand):
    # 1. Load user through repository
    user = self.user_repo.get_by_email(command.email)
    
    # 2. Create order
    order = Order(...)
    
    # 3. Orchestration: interaction logic
    # We take data from one entity and pass it to another
    if user.is_vip():
        order.apply_discount(percent=10)
    
    # 4. Save
    self.order_repo.save(order)
Important: Use Case should not calculate discount itself (`total * 0.9`). It should call method `order.apply_discount()`. Logic must remain in entities!

‚ö†Ô∏è Application Layer Antipatterns

1. Fat Use Case

This is when Use Case starts doing calculations instead of entities.

# ‚ùå BAD
total = 0
for item in items:
    total += item.price # Logic calculation leaked from Domain!
order.total = total

# ‚úÖ GOOD
order.calculate_total() # Use Case just delegates
2. Leaking Infrastructure

Passing framework or DB objects into Use Case.

# ‚ùå BAD
def execute(self, request: starlette.requests.Request): # Web binding!
    pass

# ‚ùå BAD
def execute(self, session: sqlalchemy.Session): # DB binding!
    pass
3. Returning Entities

Use Case should not return `Order` objects to the outside (to Web). It should return `OrderResultDTO`.

Why? Because Entity has behavior, and API just needs to return JSON. Plus, if you change Entity, you'll break the API contract.
