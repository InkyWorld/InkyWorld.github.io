Dependency Injection (DI)

How to assemble an application from separate classes without creating tight coupling.

1. The Problem: "Who Creates the Object?"

We have a `CreateOrderUseCase`. It needs an `OrderRepository`. Who should create the repository?

âŒ Bad: Create Inside

class CreateOrderUseCase:
    def __init__(self):
        # Tight coupling! Impossible to test.
        self.repo = SqlAlchemyRepository()

âœ… Good: Pass from Outside

class CreateOrderUseCase:
    # Dependency is "injected" (inserted)
    def __init__(self, repo: OrderRepository):
        self.repo = repo

The second option is better because the Use Case no longer knows what SQLAlchemy is. It works with an abstraction. But this raises a question: Where do we create these objects?

2. Composition Root

This is a single place in the application where we create all objects and wire them together. Usually this happens at application startup (for example, in the `main.py` file or a separate `dependencies.py` module).

[ container.py / Composition Root ]
  1. create_engine(DB_URL)
  2. make_session_factory(engine)
  3. repo = SqlAlchemyOrderRepository(session)
  4. email_service = SmtpEmailService(...)
  5. use_case = CreateOrderUseCase(repo, email_service) â€” assembly complete
â†“ FastAPI route â€” receives the ready UseCase via Depends

Important: Composition Root is a concept of a single assembly point for dependencies, not necessarily a main.py file. In real projects, assembly is often extracted to a separate module (container.py, dependencies.py, or apps/<app>/container.py), initialized at application startup (lifespan/startup hooks), or DI containers are used; this facilitates testing (provider overriding), scope management (app/request), and explicit lifecycle management of dependencies.

3. Manual Assembly Example (Pure Python)

In Python, complex DI frameworks are often not needed. Regular functions are sufficient.

# main.py or container.py

# 1. Import implementations (Infrastructure)
from infrastructure.db import create_engine, session_factory
from infrastructure.repositories import SqlAlchemyOrderRepository
from infrastructure.adapters import SmtpEmailService

# 2. Import Use Case (Application)
from application.use_cases import CreateOrderUseCase

# Global settings (can be loaded from env)
DB_URL = "postgresql://user:pass@localhost/db"
engine = create_engine(DB_URL)

# DEPENDENCY FACTORY (Composition Root)
def get_create_order_use_case() -> CreateOrderUseCase:
    # a. Create DB session
    session = session_factory(bind=engine)
    
    # b. Create adapters (Infrastructure)
    repo = SqlAlchemyOrderRepository(session)
    email_service = SmtpEmailService("smtp.gmail.com", ...)
    
    # c. Assemble Use Case (Injection)
    use_case = CreateOrderUseCase(
        repo=repo, 
        email_service=email_service
    )
    
    return use_case

4. FastAPI Integration (Depends)

FastAPI has a built-in DI system (`Depends`) that is perfect for the Composition Root role.

from fastapi import APIRouter, Depends
# Import our factory (see above)
from .container import get_create_order_use_case

router = APIRouter()

@router.post("/orders")
def create_order_endpoint(
    data: CreateOrderDTO,
    # FastAPI calls get_create_order_use_case function, 
    # retrieves the ready UseCase and puts it in the use_case variable
    use_case: CreateOrderUseCase = Depends(get_create_order_use_case)
):
    return use_case.execute(data)
Tip: If you need to manage lifecycle (for example, close DB session after request), use `yield` in the dependency function.

5. Unit of Work Integration

It's better to delegate transaction management to the Composition Root (dependency), rather than forcing the Use Case to handle commit/rollback. In web applications, it's convenient to create a UoW dependency that uses yield and guarantees commit/rollback/close when the request completes.

# container.py

# Session factory
from sqlalchemy.orm import sessionmaker
session_factory = sessionmaker(bind=engine)

# FastAPI dependency that takes UoW and manages transaction
def get_uow():
    session = session_factory()
    uow = SqlAlchemyUnitOfWork(session)
    try:
        yield uow
        # Commit is executed after successful use case / request execution
        uow.commit()
    except:
        # Rollback on error
        uow.rollback()
        raise
    finally:
        session.close()

# In the endpoint / container we simply get UoW as dependency
@router.post("/orders")
def create_order(cmd: CreateOrderCommand, uow: UnitOfWork = Depends(get_uow)):
    use_case = CreateOrderWithUowUseCase(uow)
    return use_case.execute(cmd)

# Use Case should not commit by itself
class CreateOrderWithUowUseCase:
    def __init__(self, uow: UnitOfWork):
        self.uow = uow

    def execute(self, cmd):
        order = Order(...)
        self.uow.orders.save(order)
        # Commit will happen after exiting the dependency

1. Provider

A provider is a class that groups dependency factories. The main tool is the @provide decorator.

class AppProvider(Provider):
    # scope=Scope.APP â€” creates object once for application lifetime (Singleton)
    @provide(scope=Scope.APP)
    def get_config(self) -> Config:
        return load_config()

    # scope=Scope.REQUEST â€” creates object anew for each request
    @provide(scope=Scope.REQUEST)
    async def get_connection(self, config: Config) -> AsyncIterable[Connection]:
        conn = await connect_db(config.db_url)
        yield conn  # Returns object
        await conn.close() # Executes after scope completes (Cleanup)

2. Alias

If you have an implementation and want to "bind" it to an interface without an extra function.

class DBProvider(Provider):
    # Bind interface to implementation
    repo = alias(source=PgRepo, provides=IRepo)

3. Decorate

Wrap an object "on the fly" (e.g., add logging or caching).

class LogProvider(Provider):
    @decorate
    def with_log(self, repo: IRepo) -> IRepo:
        return LoggedRepo(repo)

ğŸ¯ Scopes

Scope Description Use Case
Scope.APP Lives for the entire application lifetime. DB connection pool, HttpClient, Config.
Scope.REQUEST Lives within a single HTTP request or event. DB transaction, UnitOfWork, Current user.
Scope.SESSION Long-lived scope (e.g., user session). User cache in a bot.
