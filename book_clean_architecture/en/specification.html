Specification Pattern

Encapsulating complex business rules for validation and selection.

The Specification pattern is a way to turn a complex if condition into a separate named class. It's like a "face control": you have a clear list of criteria for who gets in.

Problem and Solution

âŒ Without Pattern (IF Soup)

Logic is smeared, hard to read.

# In a rental service
if (user.age >= 21 and 
    user.license_type == 'B' and 
    not user.has_active_loans):
    print("Can rent")

âœ… With Pattern (Clear Name)

The rule has a name and lives separately.

# Reads like human language
rule = CanRentCar()

if rule.is_satisfied_by(user):
    print("Can rent")

1. How it works (Validation)

We create a class with a single method is_satisfied_by.

class CanRentCarSpecification:
    def is_satisfied_by(self, user: User) -> bool:
        # All complex logic lives here
        has_age = user.age >= 21
        has_license = user.license_type == 'B'
        return has_age and has_license

2. Rule Constructor (Composition)

The power of the pattern is that you can combine small rules like Lego blocks using AND, OR, NOT.

is_adult = IsAdultSpec()
has_money = HasMoneySpec()

# Create a new rule from two old ones
# "Adult AND Has Money"
can_buy_beer = is_adult & has_money 

if can_buy_beer.is_satisfied_by(bob):
    sell_beer(bob)

3. Search (Filtering)

Instead of writing methods in the Repository for every possible case:

  • find_users_by_age(21)
  • find_users_by_age_and_money(21, 100)
  • find_users_active_and_adult(...)

We create one universal find method that accepts a Specification.

# Application Layer
# "Find me all users who can rent a car"
rule = CanRentCarSpecification()
users = repository.find(rule)
Note: To make this work with a database (SQL), inside the repository we need to translate Python code user.age >= 21 into SQL WHERE age >= 21. This is the hardest part, so ready-made libraries are often used.
